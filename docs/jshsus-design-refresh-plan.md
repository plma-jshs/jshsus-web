# 과구리 공개 웹 디자인 시스템

- 확정일: 2026-07-13
- 상태: 구현 기준(Approved)
- 적용 대상: `apps/web`

## 1. 제품 방향

과구리는 홍보용 랜딩 페이지가 아니라 학생이 공지, 식단, 일정, 신청과 커뮤니티를 빠르게
확인하는 정보 포털이다. 화면을 “그럴듯한 카드”로 채우기보다 실제 업무 흐름과 데이터 비교가
먼저 읽혀야 한다.

확정된 기준은 다음과 같다.

- 모든 공개 화면의 캔버스는 흰색이다. 회청색은 구획, 표 머리글, hover처럼 경계를 보조할 때만 쓴다.
- 기본 글꼴은 자체 번들한 `Pretendard Variable`이며, 시스템 sans-serif는 로딩 실패 시에만 fallback으로 사용한다.
- 의미 없는 영문 eyebrow, 장식용 통계, 중복 제목, 고정 높이 빈 카드는 만들지 않는다.
- 카드는 홈 대시보드와 실제로 묶인 작업 영역에만 쓴다. 목록, 상세, 폼은 흰 바탕과 구분선이 기본이다.
- 색과 애니메이션은 상태 이해 또는 조작 피드백에 필요한 경우에만 쓴다.
- 검색·버튼처럼 보이는 요소는 반드시 동작해야 한다.

## 2. 시각 토큰

공용 토큰은 `packages/ui/src/tokens.css`가 단일 기준이다.

| 역할             | 값        | 용도                         |
| ---------------- | --------- | ---------------------------- |
| canvas / surface | `#ffffff` | 페이지와 기본 표면           |
| surface subtle   | `#f8fafb` | 표 머리글, 보조 구획         |
| surface hover    | `#f3f5f7` | 목록 hover                   |
| surface selected | `#eef8f7` | 현재 탭, 모바일 현재 메뉴    |
| border           | `#e5e9ec` | 일반 구분선                  |
| border strong    | `#cfd6dc` | 표 머리글, 입력 경계         |
| brand            | `#087c82` | 주요 행동, 진행률, 현재 위치 |
| brand strong     | `#075f65` | hover와 강한 브랜드 텍스트   |
| text             | `#172033` | 제목과 본문                  |
| text muted       | `#667085` | 메타 정보                    |

일반 표면 그림자는 `0 1px 2px rgb(16 24 40 / 4%)`만 허용한다. mega-menu, drawer,
popover처럼 실제로 떠 있는 요소만 overlay 그림자를 쓴다. 카드 hover 시 요소를 위로 이동시키지 않는다.

## 3. 타이포그래피와 밀도

- 페이지 제목: 28–31px, 700
- 섹션 제목: 18–20px, 600 또는 700
- 본문: 15–16px, 400 또는 500
- 표와 보조 정보: 12–14px, 400 또는 500
- 사용할 굵기: 400, 500, 600, 700
- 긴 한글 제목에는 과한 음수 자간을 쓰지 않는다.
- 숫자 비교 열에는 `font-variant-numeric: tabular-nums`를 사용한다.

본문 폭은 페이지 성격에 맞춘다.

| 유형            |     최대 폭 | 예시                          |
| --------------- | ----------: | ----------------------------- |
| dashboard       |      1280px | 홈                            |
| list            | 1120–1160px | 공지, 게시판, 청원, 신청 목록 |
| document / form |   760–800px | 상세, 글쓰기, 신청서          |
| workspace       | 1180–1280px | 전체 학사일정, 분실물 그리드  |

## 4. 공통 페이지 유형

### 목록

제목과 한 줄 설명 다음에 48–56px 높이의 검색·필터 toolbar를 둔다. 별도 카드 컨테이너 없이
표 또는 행 목록이 바로 이어진다. 표에는 강한 상단선, 옅은 머리글 배경, 중앙 정렬 머리글을
사용한다. hover는 행 배경과 제목 밑줄만 바뀐다. 행 끝 꺾쇠는 두지 않는다.

### 문서 상세

breadcrumb → 제목 → 작성자·날짜·조회 같은 한 줄 메타 → 본문 → 첨부 → 행동 순서다. 본문을
둥근 카드 안에 다시 넣지 않고, 짧은 본문에도 큰 최소 높이를 주지 않는다. 댓글은 별도 카드가
아니라 본문 아래의 구분된 섹션이다.

### 폼

한 열의 label/field 구조를 사용한다. 입력, 편집기, 설정 행, 하단 행동의 경계를 구분선으로
만든다. placeholder로 label을 대체하지 않는다. primary action은 한 개만 둔다.

### 작업 공간

달력이나 이미지 목록처럼 넓이가 정보 이해에 직접 필요한 화면에만 사용한다. 넓은 흰 표면 안에서
toolbar와 본문을 구분하며, 장식 카드 여러 개로 쪼개지 않는다.

## 5. 전역 셸과 탐색

- 헤더는 68px 높이의 반투명 흰색과 1px 하단선으로 구성한다.
- 로고에는 그림자나 hover 이동을 넣지 않는다.
- 데스크톱 nav는 네 개 상위 카테고리를 같은 폭의 열로 둔다.
- 상위 카테고리에 pointer hover 또는 keyboard focus가 들어오면 전체 폭 mega-menu가 열린다.
- mega-menu의 각 열은 해당 상위 메뉴와 수평 중심 및 폭이 맞아야 하며 열 사이 경계가 끊기지 않는다.
- 활성 카테고리의 꺾쇠와 해당 열 상단선만 바뀐다. 메뉴 hover는 글자색만 바뀐다.
- `aria-expanded`, `aria-controls`, Escape 닫기와 focus 복귀를 제공한다.
- 모바일 topbar의 전체 메뉴는 오른쪽 drawer로 열고, 하단 탐색은 화면 가장자리에 붙은 평면 bar다.
- footer는 과구리, 사업자 정보, 개인정보처리방침, 저작권만 낮은 강조로 표시한다.

## 6. 홈

홈은 알림 띠 → 공지/식단/학사일정 → 게시판/청원 순서다.

- 카드 전체 hover 이동과 고정 높이를 없애고 실제 행 수에 맞춰 밀도를 결정한다.
- 공지는 중요/일반 분류 배지를 반복하지 않고 제목과 날짜를 우선한다.
- 식단은 아침·점심·저녁을 독립적으로 표시하며 없는 끼니에만 한 줄 empty text를 표시한다.
- 식단 날짜 이동은 카드 제목 행에서 날짜 양옆의 44px에 가까운 버튼으로 제공한다.
- 홈 미니 달력은 일정 제목 대신 일정 수만큼 작은 점을 표시하고 hover/focus popover를 제공한다.
- 다가오는 일정은 오늘부터 일주일 안의 항목만 3줄 공간에 표시한다. 세 건 이상이면 두 건과 `+n`이다.
- 개발 데이터는 실제 로컬 seed 행만 사용한다. 화면 전용 음수 ID나 클릭 불가능한 미리보기 항목을 주입하지 않는다.

## 7. 기능별 기준

### 공지사항

- 중요 공지 분리 카드와 중요/일반 분류를 제거한다.
- 열은 번호, 제목, 작성자, 등록일, 조회다.
- 전체 건수와 페이지는 toolbar에 작은 텍스트로만 둔다.
- 상세는 작성 부서, 작성자, 날짜, 조회를 한 줄 메타로 합친다.

### 자유게시판

- 목록은 공지와 같은 표 문법을 사용한다.
- 댓글 수는 제목 바로 뒤에 `[n]`으로 표시하고 조회와 합산하지 않는다.
- 상세 댓글은 구분선 기반이며 댓글 등록·신고 성공/오류를 live text로 알린다.
- 작성은 제한된 Tiptap editor를 사용한다. 본문 인라인 이미지는 draft 업로드 후 안정 URL로 치환한다.
- 익명 작성은 native checkbox와 정상적인 focus ring을 사용한다.

### 청원·제안

- 목록 열은 상태, 제목, 마감일, 참여 인원, 달성률이다.
- 상세에는 현재 인원/목표/달성률과 수평 진행 막대를 둔다.
- 답변은 본문 아래의 명확한 문서 섹션으로 표시한다.
- 작성은 제목, 마감일, 제한형 rich text를 사용한다. 인라인 이미지는 허용하지 않는다.

### 탐구활동서

- 의미 없는 4개 통계 strip을 제거한다.
- 상태 filter와 신청일/기간/장소/목적을 비교할 수 있는 표를 사용한다.
- 상세는 definition list와 처리 이력을 사용하며 발급 번호와 반려 사유를 구분한다.
- 신규 신청은 날짜 순서와 필수 입력을 submit 전에 검증한다.

### 분실물

- 목록은 4:3 이미지 중심 grid를 유지한다. desktop 4열, tablet 3/2열, mobile 1열을 기준으로 한다.
- 카드에는 분실/습득, 처리 상태, 제목, 장소, 날짜만 남긴다.
- 상세는 이미지와 정보를 2열로 배치하고 작은 화면에서는 1열로 바꾼다.
- 등록의 분실/습득 선택은 semantic radio group이다. 이미지 선택 후 preview·교체·삭제가 가능해야 한다.
- 업로드 후 본문 생성이 실패하면 파일 레코드와 객체를 정리한다.

### 학사일정

- 전체 화면은 넓은 workspace와 월 toolbar를 사용한다.
- desktop cell에는 일정 제목을 1–2줄까지 직접 표시한다. 홈 미니 달력만 점을 사용한다.
- mobile에서는 달력 아래 agenda로 날짜와 제목을 읽을 수 있게 한다.
- 날짜 button에는 전체 날짜와 일정 수가 포함된 accessible name을 제공한다.

### 마이페이지/상벌점

- 페이지 제목은 `마이페이지`, nav 이름은 `상벌점`이다.
- 이름 뒤에 `학생님`을 붙이지 않는다.
- 사용자 identity와 상점/벌점/합계를 간결한 3열 strip으로 표시하고 아래에 이력 표를 둔다.
- 세션, 상벌점, 부가 정보 요청의 오류를 하나의 “로그인 확인” 오류로 합치지 않는다.

### 로그인

- nav와 footer가 없는 순백 단독 화면이다.
- 400–420px 폭의 평면 폼을 쓰며 큰 카드 배경이나 홍보 문구를 두지 않는다.
- 로그인 기억하기와 비밀번호 찾기는 제출 버튼 바로 위 한 줄이다.
- 계정 만들기 안내는 제출 버튼 아래에 둔다.

## 8. 반응형과 접근성

| 범위       | 좌우 여백 | 기본 구조                    |
| ---------- | --------: | ---------------------------- |
| 0–767px    |      16px | 1열, flat bottom bar, drawer |
| 768–1199px |      24px | 1–2열                        |
| 1200px+    |   32–40px | 각 페이지 최대 폭 중앙 정렬  |

- 검증 폭은 320, 390, 768, 1024, 1200, 1440px이다.
- 모바일 표는 핵심 제목 한 줄과 작성자·날짜·조회 메타 한 줄로 재배치한다. 단순히 열을 숨겨
  의미를 잃지 않는다.
- 주요 control은 44×44px 이상이다.
- skip link, route별 document title, route 이동 시 heading focus, live mutation feedback을 제공한다.
- 모든 icon-only button에는 accessible name이 있어야 한다.
- `prefers-reduced-motion`에서는 전환과 animation을 사실상 제거한다.
- loading, empty, 401/403, 404, 5xx를 같은 메시지로 합치지 않는다.

## 9. 스타일 소유권

- token: `packages/ui/src/tokens.css`
- base/shell/home: `apps/web/src/styles/base.css`, `shell.css`, `home.css`
- shared page/table/editor/document: `page-scaffold.css`, `data-tables.css`, `editor.css`, `content-pages.css`
- domain: `petitions.css`, `activity-requests.css`, `lost-items.css`, `calendar.css`, `my-status.css`, `auth.css`
- responsive shell/home: `responsive.css`; domain responsive rule은 해당 domain 파일에 둔다.

기존의 모든 기능을 한 파일에 섞은 `detail-pages.css`는 사용하지 않는다. 새로운 화면은 먼저
공통 primitive를 사용하고 실제로 고유한 구조만 domain CSS에 추가한다.

## 10. 완료 기준

- 모든 공개 route가 흰 canvas와 동일한 셸/타이포그래피를 사용한다.
- 목록, 상세, 폼, workspace가 위 문법에 맞고 불필요한 카드 중첩이 없다.
- 로컬 mock/seed 목록의 모든 상세 링크가 열리거나 명시적으로 비활성이다.
- 390, 768, 1440px에서 가로 overflow, 잘린 조작, 콘텐츠를 가리는 고정 UI가 없다.
- keyboard, focus, Escape, label, live feedback, reduced motion을 확인한다.
- typecheck, lint, test, format check, production build를 통과한다.
- 별도 승인 전에는 배포하지 않는다.
