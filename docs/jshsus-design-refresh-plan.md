# 과구리 UI 디자인 시스템 및 개편 기준

- 작성일: 2026-07-12
- 상태: 구현 기준(Approved)
- 적용 대상: 학생용 공개 포털 `apps/web`, 운영자 콘솔 `apps/admin`

## 1. 목표와 확정 사항

과구리는 학교생활에 필요한 정보를 빠르게 훑고 바로 행동할 수 있는 **깔끔하고 모던한
정보 포털**을 지향한다. 장식적인 랜딩 페이지보다 공지, 식단, 일정, 커뮤니티와 학생 개인
상태의 우선순위를 명확히 보여주는 것이 핵심이다.

확정된 방향은 다음과 같다.

- 첨부 시안의 밝은 배경, 청록 워드마크, 낮은 대비의 그림자, 조밀한 정보 카드 구성을 따른다.
- 공개 포털 전체를 먼저 같은 언어로 개편하고, 관리자 콘솔은 같은 토큰을 사용하되 더 높은
  정보 밀도의 별도 패턴을 사용한다.
- 헤더 검색창과 테마 전환 버튼은 제거한다. 검색처럼 보이지만 동작하지 않는 장식은 두지 않는다.
- 이번 디자인은 라이트 테마 전용이다. 다크 모드는 별도 요구사항과 접근성 검증을 갖춘 뒤 추가한다.
- 식단은 NEIS 데이터를 사용하고, 학사일정은 NEIS 일정과 학교가 직접 등록한 일정을 함께 표시한다.
  실제 데이터가 준비되지 않았을 때 샘플 데이터를 사용자에게 사실처럼 보여주지 않는다.
- 무거운 UI 프레임워크와 Storybook을 추가하지 않는다. `@jshsus/ui`의 토큰과 작은 패턴 위에
  현재 React 구조를 점진적으로 정리한다.

## 2. 디자인 원칙

### 2.1 한 화면에서 중요한 것을 파악한다

홈은 홍보용 히어로가 아니라 오늘의 학교생활 대시보드다. 상단부터 알림, 공지, 오늘의 식단,
학사일정, 커뮤니티 순으로 읽혀야 한다. 각 카드는 하나의 질문에 답한다.

- 공지: 지금 반드시 알아야 할 것은 무엇인가?
- 식단: 오늘 무엇을 먹는가?
- 일정: 오늘과 가까운 날짜에 무엇이 있는가?
- 커뮤니티: 학생들이 무엇을 이야기하고 제안하는가?
- 개인 상태: 내가 처리하거나 확인할 것은 무엇인가?

### 2.2 절제된 표면을 사용한다

흰 카드와 연한 회청색 캔버스를 기본으로 한다. 큰 그라데이션, 유리 효과, 두꺼운 테두리,
과도한 그림자를 겹치지 않는다. 카드의 구분은 여백과 1px 선, 낮은 그림자 중 필요한 수단만 쓴다.

### 2.3 색은 의미가 있을 때만 사용한다

청록은 브랜드와 주요 행동, 파랑은 정보와 링크에 사용한다. 빨강·주황·초록은 상태를 전달할
때만 사용하며 장식용으로 반복하지 않는다. 색만으로 상태를 구분하지 않고 텍스트나 아이콘을
함께 제공한다.

### 2.4 공개 포털과 운영 화면의 성격을 구분한다

- 공개 포털: 편안한 읽기, 44px 이상의 조작 영역, 카드 중심, 모바일 우선 탐색
- 관리자 콘솔: 빠른 비교와 처리, 40px 기본 컨트롤, 표·필터·상태 중심, 위험 작업 명시

둘은 색과 타이포그래피를 공유하지만 화면 밀도와 컴포넌트 배치를 억지로 동일하게 만들지 않는다.

## 3. 토큰 사용 계약

디자인 토큰은 `@jshsus/ui`가 단일 기준이다.

```css
@import '@jshsus/ui/tokens.css';
@import '@jshsus/ui/public.css'; /* apps/web */
/* 또는 */
@import '@jshsus/ui/admin.css'; /* apps/admin */
```

앱에서는 참조 팔레트(`--j-ref-*`)보다 의미 토큰(`--j-color-*`, `--j-space-*`)을 우선 사용한다.
앱 전용 값이 필요하면 기존 토큰을 조합하고, 같은 값이 세 번 이상 반복되면 공용 토큰 승격을
검토한다. 공용 토큰을 화면 하나에 맞추기 위해 변경하지 않는다.

### 3.1 색상

| 역할        | 토큰                       | 기준값    | 사용처                         |
| ----------- | -------------------------- | --------- | ------------------------------ |
| 브랜드      | `--j-color-brand`          | `#087c82` | 워드마크, 주요 버튼, 현재 위치 |
| 브랜드 강조 | `--j-color-brand-strong`   | `#075f65` | hover, 강한 제목               |
| 보조 정보   | `--j-color-info`           | `#1769aa` | 일반 배지, 정보 링크           |
| 캔버스      | `--j-color-canvas`         | `#f5f8fa` | 페이지 배경                    |
| 표면        | `--j-color-surface`        | `#ffffff` | 카드, 메뉴, 모달               |
| 약한 표면   | `--j-color-surface-subtle` | `#f3f7f8` | 필터, 보조 블록                |
| 기본 글자   | `--j-color-text`           | `#172033` | 본문, 제목                     |
| 보조 글자   | `--j-color-text-muted`     | `#667085` | 메타 정보                      |
| 경계        | `--j-color-border`         | `#e3e9ee` | 카드, 행 구분                  |
| 성공        | `--j-color-positive`       | `#16805b` | 완료, 정상                     |
| 주의        | `--j-color-warning`        | `#ad5f00` | 검토, 마감 임박                |
| 위험        | `--j-color-danger`         | `#d13c34` | 중요, 오류, 삭제               |

상태의 연한 배경은 `*-subtle`, 그 위 글자는 해당 상태의 기본 색상 토큰을 사용한다. 흰 바탕 위
일반 본문은 최소 4.5:1, 큰 글자는 최소 3:1 명암비를 유지한다.

### 3.2 타이포그래피

기본 글꼴은 `Pretendard Variable`, `Pretendard`, 시스템 한글 sans-serif 순서다. Pretendard를
제공할 때는 자체 호스팅한 WOFF2와 `font-display: swap`을 사용한다. 외부 폰트 CDN을 런타임
필수 의존성으로 만들지 않는다.

| 역할        | 크기/행간                            | 굵기         |
| ----------- | ------------------------------------ | ------------ |
| 페이지 제목 | `--j-text-2xl` / `--j-leading-tight` | 700          |
| 카드 제목   | `--j-text-xl` / `--j-leading-tight`  | 700          |
| 섹션 제목   | `--j-text-lg` / `--j-leading-snug`   | 700          |
| 본문        | `--j-text-md` / `--j-leading-normal` | 400 또는 500 |
| 보조 정보   | `--j-text-sm` / `--j-leading-normal` | 400 또는 500 |
| 작은 배지   | `--j-text-xs` / `--j-leading-snug`   | 600          |

한글 본문에 자간을 과하게 줄이지 않는다. 숫자 열은 필요한 경우
`font-variant-numeric: tabular-nums`를 사용한다.

### 3.3 간격, 크기, 모서리

- 간격은 4px 기반의 `--j-space-1`부터 `--j-space-16`까지 사용한다.
- 공개 화면 본문 최대 폭은 `--j-layout-public-max: 1440px`다.
- 관리자 본문 최대 폭은 `--j-layout-admin-max: 1600px`다.
- 공개 포털 기본 컨트롤 높이는 `--j-control-public: 44px`, 관리자는
  `--j-control-admin: 40px`다.
- 카드 기본 모서리는 `--j-radius-lg: 14px`, 입력과 버튼은 `--j-radius-md: 10px`,
  배지는 `--j-radius-pill`을 사용한다.
- 카드 안에 같은 크기의 카드를 반복 중첩하지 않는다.

### 3.4 그림자와 움직임

- 일반 카드는 `--j-shadow-card`, 떠 있는 메뉴와 모달은 `--j-shadow-overlay`를 사용한다.
- hover만으로 카드 전체가 크게 이동하거나 확대되지 않는다.
- 전환은 색·테두리·그림자에 한해 `--j-motion-fast` 또는 `--j-motion-base`를 사용한다.
- `prefers-reduced-motion: reduce`에서는 애니메이션과 부드러운 스크롤을 제거한다.

## 4. 반응형 기준

CSS 미디어 쿼리의 기준값은 문서와 패키지 런타임 상수에 함께 명시한다. CSS custom property는
미디어 쿼리 조건에 사용할 수 없으므로 숫자를 임의로 변형하지 않는다.

| 범위    | 폭           | 기본 구성                                           |
| ------- | ------------ | --------------------------------------------------- |
| Mobile  | `0–767px`    | 1열, 16px 좌우 여백, 핵심 메뉴를 하단/드로어로 제공 |
| Tablet  | `768–1199px` | 2열, 24px 좌우 여백, 긴 표는 카드 또는 수평 스크롤  |
| Desktop | `1200px+`    | 홈 3열, 32–40px 여백, 최대 폭 중앙 정렬             |

320px에서도 가로 페이지 스크롤이 생기지 않아야 한다. 모바일에서 데스크톱 표를 단순 축소하지
말고, 중요한 필드를 앞에 두거나 행을 카드 형태로 바꾼다. 하단 고정 탐색이 있으면 본문과 footer에
안전 여백을 확보한다.

## 5. 공개 포털 패턴

### 5.1 전역 셸

- 헤더 왼쪽에는 `과구리` 워드마크와 학교 포털 설명을 둔다.
- 데스크톱 중앙에는 `소식·일정`, `학교생활`, `커뮤니티`, `방송·도구`를 둔다.
- 오른쪽에는 로그인 또는 간결한 사용자 메뉴만 둔다. 검색창과 테마 전환은 두지 않는다.
- 모바일에서는 워드마크, 알림/사용자 행동, 메뉴 진입점을 한 줄에 유지한다.
- footer에는 학교 저작권 문구와 현재 실제 경로가 있는 공지사항, 분실물, 관리자 링크를 제공한다.
  개인정보처리방침과 이용약관은 학교가 확정한 원문과 담당 문의 경로를 받은 뒤 별도 페이지로 추가한다.

### 5.2 홈 정보 구조

Desktop에서는 다음 순서를 기준으로 한다.

1. 전체 폭 알림 띠: 고정 공지 한 건과 상세 이동
2. 상단 3열: 공지사항 / 오늘의 식단 / 학사일정
3. 하단 2열: 자유게시판 / 청원·제안
4. 로그인 사용자 보조 영역: 처리할 활동 신청과 개인 상태. 내용이 없으면 홈을 밀어내는 빈 카드를
   만들지 않는다.

Tablet은 상단 정보를 2열로 재배치하고 일정 또는 식단을 전체 폭으로 확장할 수 있다. Mobile은
알림 → 중요 공지 → 식단 → 일정 → 개인 상태 → 커뮤니티 순으로 한 열에 배치해 DOM과 시각 순서를
일치시킨다.

데이터 상태는 다음과 같이 구분한다.

- 로딩: 실제 카드 높이와 비슷한 skeleton. 회전 spinner만 가운데 두지 않는다.
- 없음: “등록된 식단이 없습니다”처럼 사실을 짧게 설명하고 필요한 행동만 제공한다.
- 오류: 카드 전체를 무너뜨리지 않고 재시도와 마지막 갱신 여부를 표시한다.
- 미연동: 샘플 대신 “학교 정보 연동 준비 중”으로 표시한다.

식단은 조식·중식·석식과 알레르기 정보 진입점을 제공한다. 학사일정은 달력과 가까운 일정 목록을
함께 사용하며, NEIS와 학교 등록 일정의 출처를 상세 또는 접근 가능한 레이블로 구분한다.

### 5.3 목록·상세·폼

- 목록: 페이지 제목/설명 → 필요한 필터와 주요 행동 → 결과 수 → 목록 순서다.
- 목록 행은 제목이 가장 강하고, 작성자·날짜·조회·댓글은 한 단계 약하게 보인다.
- 상세: 뒤로가기 breadcrumb → 제목과 상태 → 메타 → 본문 → 첨부 → 관련 행동 순서다.
- 폼: 한 열을 기본으로 하며 label을 placeholder로 대체하지 않는다. 저장 버튼은 폼 끝에서 명확한
  동사를 사용한다.
- 모바일에서는 주요 작성/신청 행동을 쉽게 찾을 수 있게 하되 콘텐츠를 가리는 고정 버튼을 남용하지
  않는다.

### 5.4 상태와 피드백

배지는 `neutral`, `info`, `positive`, `warning`, `danger` 다섯 톤을 사용한다. 서비스별로 임의의
색을 추가하기보다 아래 의미를 먼저 맞춘다.

- neutral: 초안, 종료, 일반 분류
- info: 일반 공지, 접수, 진행 정보
- positive: 승인, 답변 완료, 습득/매칭 완료
- warning: 검토 중, 마감 임박, 응답 대기
- danger: 중요 공지, 반려, 오류, 삭제

성공 메시지는 다음 화면 이동 후에도 필요한 경우 toast나 인라인 알림으로 한 번 확인할 수 있어야
한다. 오류는 사용자 탓으로 표현하지 않고 복구 행동을 함께 제공한다.

## 6. 관리자 콘솔 패턴

관리자 화면은 공개 포털의 카드를 복제하지 않고 처리 흐름을 중심으로 구성한다.

- 고정 sidebar는 `요약`, `생활 관리`, `커뮤니티`, `운영` 그룹을 유지하되 현재 메뉴를 청록 표면과
  왼쪽 강조선으로 명확히 표시한다.
- topbar에는 현재 페이지, 처리 대기 수, 운영자 역할, 로그아웃을 둔다.
- 페이지는 제목/설명/주요 행동 → 요약 지표 → 필터 toolbar → 데이터 순서다.
- 표 머리글은 sticky를 허용하고, 긴 목록은 pagination 또는 서버 필터를 사용한다.
- 표의 zebra stripe보다 hover와 1px 행 구분을 우선한다.
- 파괴적 작업은 위험 색, 구체적 동사, 대상 이름이 포함된 확인 절차를 사용한다.
- 빈 상태와 권한 없음은 구분한다. 권한이 없는 기능을 오류처럼 보이게 하지 않는다.

공개 포털보다 조밀하지만 40px 이하의 핵심 조작 영역을 만들지 않는다. 표 안에 여러 아이콘만
나열할 때는 각각 accessible name과 tooltip을 제공한다.

## 7. 접근성 기준

- 모든 화면은 키보드만으로 논리적 순서에 따라 이동할 수 있어야 한다.
- `:focus-visible`은 `--j-focus-ring`을 사용하고 제거하지 않는다.
- 아이콘 단독 버튼에는 `aria-label`, 장식 아이콘에는 `aria-hidden="true"`를 적용한다.
- 알림/오류는 색뿐 아니라 텍스트와 적절한 live region으로 전달한다.
- 입력 오류는 해당 필드와 프로그램적으로 연결한다.
- 현재 메뉴는 `aria-current="page"`, 펼침 메뉴는 `aria-expanded`를 사용한다.
- heading 단계는 시각 크기와 무관하게 건너뛰지 않는다.
- 최소 터치 영역은 공개 44×44px, 관리자 40×40px다.
- 확대 200%와 모바일 320px에서 기능과 내용을 잃지 않아야 한다.

## 9. Do / Don't

### Do

- 제목, 메타, 행동의 시각적 우선순위를 여백과 굵기로 먼저 만든다.
- 같은 상태는 어느 페이지에서도 같은 톤과 용어를 사용한다.
- 카드 전체 클릭 영역과 내부 링크가 충돌하지 않게 semantic HTML을 사용한다.
- 실제 긴 한글 제목, 데이터 없음, 오류, 권한 없음까지 디자인한다.
- 모바일 순서를 DOM 순서와 맞춰 화면낭독기 흐름을 보존한다.

### Don't

- 검색·필터·버튼처럼 보이는 동작하지 않는 장식을 두지 않는다.
- 샘플 식단이나 일정을 실제 운영 데이터처럼 노출하지 않는다.
- 모든 정보를 각각 다른 색 카드로 만들지 않는다.
- 작은 회색 글자만으로 중요한 상태나 오류를 전달하지 않는다.
- 한 카드 안에 카드, 그 안에 다시 배지를 무한 중첩하지 않는다.
- 공개 화면에 관리자용 조밀한 표를 그대로 노출하지 않는다.
- inline style과 임의 hex 색상을 반복해 토큰 계약을 우회하지 않는다.

## 10. 구현 순서와 완료 기준

1. `@jshsus/ui` 토큰 적용 및 기존 중복 변수 제거
2. 공개 전역 셸과 홈 정보 대시보드 개편
3. 공개 목록·상세·폼과 로그인·개인 상태 개편
4. NEIS 식단·학사일정 및 학교 일정 관리 기능 연결
5. 관리자 콘솔에 관리자 패턴 적용
6. 반응형·키보드·상태별 QA

완료 조건은 다음과 같다.

- 공개 포털 전 페이지가 같은 header, 폭, 타이포그래피, 버튼, 상태 언어를 사용한다.
- 홈에서 공지, 오늘의 식단, 학사일정, 커뮤니티를 한 화면 흐름으로 확인할 수 있다.
- 검색창과 테마 전환처럼 범위에서 제외된 컨트롤이 남아 있지 않다.
- 320px, 768px, 1200px, 1440px에서 핵심 정보가 겹치거나 잘리지 않는다.
- 키보드 포커스, 명암비, label, 오류 전달 기준을 만족한다.
- 로딩/없음/오류/권한 없음 상태에 가짜 데이터나 깨진 빈 카드가 없다.
- 관리자 화면은 같은 브랜드를 유지하면서 공개 포털보다 처리 효율이 높은 밀도를 갖는다.
