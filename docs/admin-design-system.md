# 과구리 관리자 UI 운영 기준

관리자 화면은 학생 포털의 시각 언어를 공유하되, 장식보다 빠른 조회와 안전한 처리를 우선한다. 새 화면은 이 문서를 기본값으로 사용하고, 예외가 필요하면 해당 업무 흐름에서 이유가 드러나야 한다.

## 1. 화면 원칙

- 본문 캔버스는 `#f7faf9`, 실제 데이터 영역은 흰색 표면으로 구분한다.
- 한 화면에 같은 의미의 요약 카드와 표를 중복하지 않는다.
- 제목 아래에는 사용자가 지금 결정하는 데 필요한 설명만 둔다. 기능 이름을 문장으로 반복하는 문구, 구현 예정 안내, 인수인계용 운영 원칙은 화면에서 제거한다.
- 목록 업무는 `필터/검색 → 표 → 페이지네이션` 순서로 구성한다. 표를 여러 카드로 쪼개지 않는다.
- 생성과 수정은 페이지 안에 긴 입력 카드를 상시 노출하지 않고 Dialog 또는 Drawer에서 처리한다. 위험 작업은 별도 확인 Dialog를 사용한다.
- 성공은 Toast로 짧게 알리고, 검증 오류는 해당 필드 가까이에 표시한다.

## 2. 공통 시각 규칙

- 글꼴: Pretendard Variable
- 본문 기본 크기: 14px, 표 헤더: 14px/600, 페이지 제목: 24px~28px
- 입력, 선택, 기본 버튼 높이: 40px
- 패널 모서리: 12px, 입력과 버튼 모서리: 8px
- teal은 서비스 브랜드와 내비게이션에 사용하고, 저장·등록·적용 같은 primary 작업은 blue 계열을 사용한다. 위험 작업과 검증 오류는 red 계열을 사용한다.
- 애니메이션은 Dialog/Drawer의 짧은 등장, 버튼·행 hover 정도로 제한한다. 요소 이동이나 과도한 확대는 사용하지 않는다.

## 3. 표

- 표와 행 배경은 흰색, 헤더는 옅은 회색 표면과 상단 강조선으로 분리한다.
- 헤더 텍스트는 중앙 정렬한다. 본문은 텍스트 왼쪽, 숫자·상태·날짜 중앙, 금액이나 합계는 오른쪽 정렬할 수 있다.
- 빈 결과도 헤더와 한 개의 빈 상태 행을 유지한다.
- 페이지가 나뉘는 모든 표에는 숫자형 페이지네이션을 둔다. 서버 페이지네이션이 가능한 API는 서버 기준 total/page/pageSize를 사용한다.
- 페이지가 나뉘는 표는 기본 20건이며 20·50·100건을 선택할 수 있다. 정렬 가능한 열은 헤더 클릭으로 오름차순·내림차순을 전환한다.
- 짧은 고정 값 열은 `meta.widthPreset`을 사용한다. `selection`(52px), `index`(68px), `short`(96px), `status`(108px), `action`(88px)을 제공하며 본문 값은 기본 중앙 정렬된다. 예외가 필요한 열만 `meta.align`으로 덮어쓴다.
- 행 작업 열의 헤더는 `작업`으로 통일한다. 정렬 중인 화살표와 헤더 텍스트는 진한 blue, 비활성 화살표는 옅은 gray로 구분한다.
- 검색, 날짜 범위, 업무 필터, 표시 건수와 주요 생성 버튼은 데스크톱에서 하나의 툴바 행에 둔다. 좁은 화면에서만 줄바꿈한다.
- 행 hover는 옅은 회색 배경으로만 표시한다. 행이나 꺾쇠가 이동하는 효과는 사용하지 않는다.
- 행 작업은 마지막 `작업` 열에 아이콘 버튼으로 배치하고 접근성 이름을 제공한다. 기본 작업은 blue, 위험 작업은 red를 사용하며 상세 정보가 많으면 Drawer를 연다.

## 4. 폼과 선택

- 라벨은 입력 위에 배치하고 placeholder를 라벨 대신 사용하지 않는다.
- 한 행에 모든 입력을 억지로 넣지 않는다. 관련 필드만 2~3열 grid로 묶고, 모바일에서는 1열로 접는다.
- 체크박스는 브라우저 기본 크기를 무리하게 늘리지 않고 라벨과 1개 행으로 정렬한다.
- 빈번한 사용자 선택은 별도 Dialog를 열지 않고 검색 입력과 결과 목록이 결합된 combobox를 기본으로 사용한다. 학년·반·번호 직접 입력도 같은 폼 안에 둔다.
- 학생 검색은 학번과 이름을 모두 지원하며 결과에는 `학번 이름`만 표시한다. 교직원 식별번호는 사이트가 발급한 숫자 6자리만 사용한다.
- 학년·반·번호처럼 값이 짧은 입력은 내용에 맞는 고정 너비를 사용하고, 검색 입력도 필요한 범위까지만 넓힌다. 남는 화면 너비를 입력 요소로 억지로 채우지 않는다.

## 5. 정보 구조

- 학생·교직원: 계정과 인적 프로필의 생성, 조회, 상태 변경, 비밀번호 초기화, 사용자별 역할 배정
- IAM 권한: 역할 정의와 permission 조합 관리. 사용자 목록이나 계정 생성 기능은 두지 않는다.
- 상벌점: 현황, 기록, 부여, 사유 관리, 퇴사·학기 조정으로 분리한다.
- 탐구활동서: 전체 현황과 승인·발급 업무를 분리한다.
- 기숙사: 운영 현황과 배정·민원 처리 업무를 분리한다.
- 기타: 휴대폰 보관함과 기상곡처럼 서로 독립적인 운영 도구를 둔다.
- 사이트 운영: 공지, 자유게시판, 분실물, 학사일정을 관리한다. 청원은 학생회 소관이므로 관리자 포털에서 제외한다.
- 관리자: 감사 로그와 실제 서비스 상태만 제공한다. 개발 문서나 운영 원칙은 `docs/`에 둔다.

## 6. 주요 업무 규칙

- 상벌점은 상점·벌점·기타 원장을 합산한 순합계로 판단한다.
- 순합계 `-10`점부터 `-19`점은 퇴사 위험, `-20`점 이하는 퇴사 대상으로 표시한다.
- 퇴사 시 과거 원장을 삭제하지 않고 시스템 조정 원장을 추가해 순합계를 0점으로 만든다.
- 새 학기 반감은 기존 원장을 수정하지 않고 상점 누계와 벌점 누계를 각각 절반으로 내림한 차이를 시스템 조정으로 기록한다.
- 시스템 자동 작업은 사람이나 테스트 계정을 재사용하지 않고 별도 시스템 주체와 감사 로그로 남긴다.
- 학사일정은 전남과학고 홈페이지 일정을 읽기 전용으로 표시하고, 학교 자체 일정만 생성·수정·공개 전환·삭제할 수 있다.
- 탐구활동서는 대표 학생 한 명과 한 명 이상의 참가 학생, 장소, 목적, 활동 시간을 보존한다. 관리자 인쇄는 선택 행 인쇄가 아니라 오늘 활동하는 승인된 탐구활동서 전체 일괄 인쇄를 기본으로 한다.
- 기숙사 배정은 남학생 송죽관·여학생 동백관을 분리하고 방별 `capacity`와 권장 `grade`를 따른다. 방 식별자는 `생활관 + 호실명` 조합이다. 같은 방에는 같은 반 학생을 중복 배정하지 않으며, 해당 학기 룸메이트 블랙리스트의 어느 방향 관계도 허용하지 않는다.
- 기숙사 추첨 미리보기는 DB를 변경하지 않는 편집용 결과다. 적용 시 최신 방 정원·성별·학년·반·블랙리스트를 다시 검증한 뒤 한 트랜잭션으로 저장한다.
- 방 상벌점은 방을 선택하면 현재 배정 인원을 목록에 한 번에 추가하고, 적용 전에 개별 학생을 제외하거나 점수·사유를 수정할 수 있다.

## 7. 반응형 기준

- 데스크톱에서는 고정 사이드바와 넓은 데이터 표를 사용한다.
- 좁은 화면에서는 사이드바를 Drawer로 전환하고, 필터는 1열 또는 2열로 접는다.
- 표의 핵심 열은 유지하고 부가 열은 가로 스크롤로 제공한다. 모바일 카드 목록으로 임의 변환해 정보 구조를 바꾸지 않는다.
